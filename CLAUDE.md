# CLAUDE.md — МемКат

## Роль
Product design engineer. Будуємо новий продукт з нуля.

## Контекст команди й ресурсів
- Команда: одна людина (дизайнер), весь продукт на ній.
- Бюджет на інфраструктуру: ~$0. Немає коштів на сервери й хмари.
- Темп: треба швидко до першого публічного релізу (жорсткого дедлайну немає).
- **Наскрізний принцип:** усе, що вимагає сервера, команди або грошей — відкладаємо. MVP має працювати на $0 інфраструктури.

---

## Продукт

**Ідея:** МемКат — відеоредактор для блогерів із вбудованою бібліотекою мемів.

**Мета:** прибрати рутину, коли монтуєш ролик і доводиться шукати меми, звуки й тренди по інших застосунках — усе має бути під рукою в одному вікні.

**Аудиторія:** контент-мейкери для TikTok / YouTube Shorts / Reels, 16–30, монтують швидко й часто, цінують швидкість і актуальні тренди.

**Позиціонування:** не «ще один редактор», а **найшвидший спосіб зробити мем-відео**. Усе, що не працює на цю тезу — ріжемо.

---

## Платформа
- Веб, **mobile-first** (для максимального охвату й бо аудиторія монтує з телефону).
- Працює так само на десктопі/планшеті через адаптив — для довших проєктів.
- **Імпорт відео — з будь-якого пристрою:** телефон, ПК, ноут, планшет. Не привʼязуємось лише до мобільного завантаження.

---

## MVP (v1)

Мінімальний конкурентний набір. Полір важливіший за кількість фіч — беремо рівно стільки, щоб петля «зняв → додав мем → виклав» працювала бездоганно:

- імпорт відео з будь-якого пристрою (телефон / ПК / ноут / планшет);
- таймлайн з базовим монтажем: обрізка, розрізання, зміна порядку кліпів;
- накладання мема / стікера з бібліотеки поверх відео;
- додавання звуку;
- текст / субтитри вручну;
- експорт: 9:16 (TikTok / Reels / Shorts) і 1:1 (Instagram feed). Десктопний 16:9 — не в MVP: інша ніша, інший юзер, тисне на серверний рендер і розмиває позиціонування.

**Свідомо НЕ робимо в v1:** мультитрек зі складною композицією, кейфрейми / анімацію, кольорокорекцію, авто-субтитри (AI), колаборацію, акаунти. Це фічі наступних версій.

**Конкуренти:** CapCut, Canva, Clideo. Не змагаємось із глибиною редактора CapCut — програємо. Єдина чітка відмінність: **меми й тренди вбудовані, нуль перемикань між застосунками.**

## Технічні обмеження для рендеру відео
- Прев'ю в редакторі — на клієнті через Canvas/WebGL + Web Audio (реал-тайм, без сервера).
- Експорт — гібридний підхід: основний шлях через **WebCodecs** (апаратне кодування на пристрої, швидко й безкоштовно), fallback на серверний рендер (FFmpeg) для старих браузерів або слабких пристроїв — **відкладений, бо вимагає сервера**.
- Закласти **нормалізацію різнорідних кліпів** (різні кодеки, fps, роздільності з бібліотеки мемів) в єдину часову шкалу ще на рівні архітектури — це найбільший технічний ризик.
- ffmpeg.wasm не використовувати як основний движок — лише як можливий запасний варіант.
- **Перший технічний прототип = експорт.** До полірування UI зробити: кліп + мем + звук → робочий .mp4 через WebCodecs. Якщо це стабільно працює — решта діло техніки.

## Бібліотека контенту

**Джерела (своєї бази немає):** офіційні API — **Giphy і Tenor** (меми, GIF, стікери; безкоштовний tier, ліцензування на їхньому боці). Звук — CC0 / royalty-free: Pixabay, Mixkit, Free Music Archive (фільтр CC0).

**Принцип:** не «чим більше, тим краще», а «чим менше юридичного ризику, тим краще».

**Заборонено:** краулінг TikTok / YouTube. Порушення їхніх правил + копірайт-міна.

**Тренди:** без ручної модерації (немає команди). Беремо **trending endpoint** Giphy/Tenor — вони вже рахують популярне. Нуль роботи з нашого боку.

**Ліцензування:** тільки відкритий доступ + офіційні API. Виконати вимоги атрибуції (напр. «Powered by GIPHY»), інакше доступ відключать. Перед запуском перевірити Terms кожного джерела.

## Збереження
- Усе **локально в браузері через IndexedDB**: проєкти, налаштування, імпортовані відео. $0, офлайн, без акаунтів.
- Без хмари й без акаунтів у MVP.
- Чесні мінуси: проєкт привʼязаний до одного пристрою/браузера, може злетіти при чистці кешу. Синхронізація між пристроями — пізніше, якщо зʼявиться бюджет на бекенд.

## Мови
- База: **українська + англійська.**
- Зробити **i18n з першого дня** (усі тексти в окремих файлах) — тоді нова мова = переклад, а не переписування.
- Розширення (коли буде трафік): португальська (Бразилія), іспанська — найбільші ринки креаторів короткого відео. Соло — не розпорошуватись завчасно.

---

## Бізнес

**Монетизація:** старт **повністю безкоштовно** (зібрати юзерів і фідбек), монетизація — потім. Схема freemium: безкоштовно з вотермаркою; прибрати вотермарку + преміум-меми + HD-експорт = платно. Реалізм: аудиторія 16–30 платить погано — не закладати на це бізнес-план рано.

**Ринок запуску:** продукт глобальний/англомовний (веб не має бар'єрів дистрибуції), але **перші юзери й фідбек — з України**, де є мережа й канали. Далі органічне розширення англійською. Окремий фокус на СНД не робимо.

**Бюджет на запуск:** сервер для MVP **не потрібен**. Рендер на клієнті (WebCodecs) + локальне сховище (IndexedDB) + безкоштовні API + безкоштовний хостинг статики (Cloudflare Pages / Vercel / Netlify). Реальні витрати — лише домен (~$10–15/рік). Сервер зʼявляється тільки під fallback-рендер, і це відкладено.

## Робочий процес із моделлю
- Прототип не робимо, поки не буде команди «починаємо».
- Спочатку — архітектура, потім — реалізація.

## Правило: MD → HTML

Будь-які дані, висновки, оновлення статусів — спочатку в MD-файл, потім одразу в HTML.

**Який MD-файл:**
- `research/research.md` — нові дані ресерчу, висновки, аудит тверджень, питання
- `research/personas.md` — будь-що що стосується Ані, Дениса, Лізи
- `research/jtbd.md` — статуси jobs, матриця, висновок матриці

**Після запису в MD** — одразу оновити відповідну секцію в `research/research.html` (без окремої команди).

**Якщо нова інфа не лягає в жоден MD-файл або жодну секцію HTML** — запропонувати: в який файл додати і яку нову секцію в HTML створити. Не записувати нікуди без підтвердження.

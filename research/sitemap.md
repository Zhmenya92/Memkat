### 1. Проект (Project)

**Що це:** Контейнер усієї монтажної сесії — зберігається в IndexedDB локально, без акаунту, без сервера.

**Поля:**
- `id` — унікальний ключ у IndexedDB
- `[?] назва` — автоназва (дата/час) або [?] задана юзером
- `кліпи[]` — впорядкований список Відео-кліпів
- `оверлеї[]` — список МемОверлеїв
- `формат_виходу` — "9:16" | "1:1"
- `дата_змінено` — для сортування в списку проектів [?]

**Job що породжує:**
- **Main** — "зняв → додав мем → виклав": Проект є контейнером цієї петлі; без нього немає куди зберегти стан між кроками
- **R5** — "зберегти у правильному форматі": `формат_виходу` зберігається тут
- **E1** — "відчути що процес не поборов": Проект — єдиний об'єкт що переживає сесію і дозволяє повернутись

**Зв'язки:**
- Проект ↔ Відео-кліп (1 : many)
- Проект ↔ МемОверлей (1 : many)

---

### 2. Відео-кліп (VideoClip)

**Що це:** Один відеофрагмент — імпортований файл або його частина після обрізки. Лежить на таймлайні в певному порядку.

**Поля:**
- `blob_ref` — посилання на blob у IndexedDB (сам файл)
- `тривалість` — тривалість у мс (після обрізки)
- `in_point` — початок вирізки з оригінального файлу
- `out_point` — кінець вирізки
- `порядок` — позиція на таймлайні
- `[?] codec_info` — метадані кодеку/fps/роздільності для нормалізації в WebCodecs

**Job що породжує:**
- **Main** — "зняв": відзнята відеосировина — перший крок петлі; без кліпів немає чого монтувати

**Зв'язки:**
- belongs to → Проект

---

### 3. GIF / Мем (Library GIF)

**Що це:** Один GIF-елемент із зовнішньої бібліотеки (Giphy або Tenor). Не належить МемКат — зберігаємо тільки метадані + посилання.

**Поля:**
- `id` — ідентифікатор у Giphy або Tenor
- `provider` — "giphy" | "tenor"
- `url_preview` — WebP/маленький GIF для відображення в пікері
- `url_full` — повний GIF для вставки і рендеру
- `width`, `height` — оригінальні розміри (px)
- `[?] duration` — тривалість анімації (мс)
- `[?] tags` — теги з API, якщо кешуємо для mood-search

**Job що породжує:**
- **R1** — "знайти мем за настроєм/емоджі": GIF — шуканий об'єкт; emoji/text query → список GIF
- **R2** — "бачити актуальний контент з нуля": trending endpoint повертає масив GIF без запиту
- **R3** — "вставити мем не виходячи з монтажу": inline picker оперує GIF як одиницею вибору
- **R4** — "повторно знайти мем без нового пошуку": Нещодавні кешують GIF objects

**Зв'язки:**
- referenced by ← МемОверлей (placed instance)
- referenced by ← Нещодавні (cache)

---

### 4. МемОверлей (MemeOverlay)

**Що це:** Один розміщений GIF поверх відео в конкретний момент часу. Відрізняється від GIF-з-бібліотеки: GIF — «що вибрали», МемОверлей — «куди і коли вставили».

**Поля:**
- `gif_id` — ідентифікатор GIF
- `gif_provider` — "giphy" | "tenor"
- `start_time` — момент появи на таймлайні (мс від початку проекту) **Правило: = позиція playhead у момент натискання «Додати мем». Якщо playhead = 0 і таймлайн порожній — start_time = 0.**
- `[?] duration` — скільки показується; якщо не задано — тривалість оригінального GIF або до кінця кліпу
- `x`, `y` — позиція поверх кадру (нормалізована 0–1)
- `width`, `height` — розмір оверлею (нормалізований 0–1)

**Job що породжує:**
- **Main** — "додав мем": МемОверлей — результат другого кроку петлі "зняв → **додав мем** → виклав"
- **R3** — "вставити мем не виходячи з монтажу": inline picker завершується створенням МемОверлею; якщо picker вибиває з потоку — цей об'єкт не виникне взагалі

**Зв'язки:**
- belongs to → Проект
- references → GIF (many МемОверлеїв → один GIF)

---

### 5. Нещодавні GIF (RecentGIFs)

**Що це:** Локальний кеш GIF, які юзер нещодавно переглянув або використав. Живе в IndexedDB. Не сервер, не акаунт — прив'язаний до браузера.

**Поля:**
- `items[]` — масив GIF objects (орієнтовний ліміт ~50, верхня межа IndexedDB-бюджету)
- `[?] viewed_at` — timestamp для кожного item, щоб витісняти старі

**Job що породжує:**
- **R4** — "повторно знайти мем без нового пошуку": Нещодавні — єдина можливість миттєво потрапити до вже переглянутого GIF без повторного пошуку

**Зв'язки:**
- references → GIF (many-to-many за змістом)

### ? Звук (AudioTrack)

**Що це:** Аудіофайл або фрагмент, накладений на відео.

**Поля (якщо реалізувати):**
- `blob_ref` / `url` — джерело (local file або CC0 URL)
- `start_time` — момент початку
- `[?] duration` — тривалість кліпу
- `[?] volume` — гучність

**Job:** Жоден у матриці. У JTBD-матриці висновок: "Додавання звуку — кандидат на виліт або відкладення."

**Чому під питанням, а не на викид:** CLAUDE.md включає звук у MVP-список. Можливо, закриває ширшу "зробити відео" петлю, але окремого job ще не сформульовано. Потребує або свого job, або явного рішення не робити.

### ? Текст / Субтитри (TextOverlay)

**Що це:** Текстовий блок поверх відео — заголовки, підписи, ручні субтитри.

**Поля (якщо реалізувати):**
- `текст` — рядок
- `[?] font_size`, `[?] колір`
- `start_time`, `[?] duration`
- `x`, `y` — позиція

**Job:** Жоден у матриці. У JTBD-матриці висновок: "Текст / субтитри — кандидат на виліт або відкладення."

**Чому під питанням:** CLAUDE.md включає "текст вручну" в MVP. Можливо, частина загального "мем-відео" потоку (text meme поверх кліпу), але ні Аня, ні Ліза, ні Денис не мають явного job що його породжує.

## Навігаційна модель

*21.06.2026 · база: sitemap.md + jtbd.md (jobs) + personas.md*
*Принцип: кожен пункт nav = вхід у job-кластер. Жодного пункту «бо так у всіх».*

---

### Три шари навігації

МемКат — task-completion app, не browsing app. Немає незалежних "режимів" між якими переключаються довільно. Навігація = лінійний pipeline: **Project → Edit → Export**. Тому замість tab bar — три функціональні шари.

---

#### GLOBAL — не потрібна в МемКат

МемКат не потребує глобальної навігації. Global nav (tab bar / always-visible nav bar) — патерн для browsing-аппів з **незалежними доменами** між якими переключаються довільно (Feed / Explore / Profile). У МемКат немає таких доменів — є лінійний pipeline: **Project List → Editor → Export**.

**Коли global мала б сенс:** якщо б продукт додав Feed актуального контенту, Explore-розділ поза монтажем, або Profile/Community — незалежні зони де юзер перебуває поза поточним завданням монтажу.

**Що замінює global:** стандартний platform navigation header (iOS/Android navigation stack). Кожен pushed-екран отримує Back автоматично — це не продуктове рішення, це конвенція ОС.

Елементи що помилково класифікувались як Global — переміщено в Contextual:

| Елемент | Де фактично | Чому не Global |
|---|---|---|
| **← Назад** (platform Back, header ліво) | Platform convention (iOS/Android надають автоматично) | Конвенція ОС, не дизайн-рішення. Закриває E1 (відчуття контролю), але це надається платформою на кожному pushed-екрані без жодного продуктового рішення. |
| **Назва + статус збереження** (header центр) | Хедер Editor | Тільки Editor: поза ним питання «де я і чи збережено» не виникає. Не global — contextual до Editor. |
| **Кнопка Експорт** (header право) | Хедер Editor, якщо є ≥1 кліп | На Project List нема що експортувати. На Export-екрані — безглуздо показувати знову. Не global — contextual до Editor. |

**Висновок:** Мем-пікер (унікальний диференціатор) = **3 тапи** для Ані. Завершення Main job (exported file) = **6 тапів** (Path A) / **5 тапів** (Path B). Переструктурування не потрібне — глибина прийнятна для обох цілей.

---

### Чому Project List як home, а не Editor

Job **E1** ("відчути що процес не поборов"): юзер знає де їхні незакінчені проекти і може повернутися. Project list = explicit safe state, "точка приземлення".

Editor як home = контекст без landing zone, тривога "що відбувається з моїм попереднім відео".

**Для Ані з existing project:** 1 тап до Редактора — це не штраф, це оптимальний шлях.

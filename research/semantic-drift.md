# Семантичні зміщення — МемКат

*Дата: 21.06.2026*
*Метод: крос-аудит jtbd.md + personas.md + sitemap.md + research.html*
*Мета: знайти де оригінальний Main job "зробити і викласти поки мем живий" спотворився або стиснувся на наступних фазах*

---

## Канонічна петля (еталон)

**Main job (jtbd.md):**
> «Коли я бачу мем або тренд у стрічці, я хочу швидко зробити і викласти своє відео з ним, щоб потрапити в момент поки він живий.»

**Три кроки петлі (CLAUDE.md):**
> «петля "зняв → додав мем → виклав" має працювати бездоганно»

Ключові параметри канонічного job:
- **Тригер:** побачила мем/тренд у стрічці (зовнішній, не "хочу змонтувати")
- **Завершення:** викласти (опублікувати, файл потрапив на платформу) — не "вставила мем"
- **Критерій успіху:** потрапити в момент поки тренд живий (часовий вимір)
- **Кроки рівноправні:** зняв = додав мем = виклав; жоден не важливіший за інший

## Зміщення 1 — "виклав" зник з кроків, залишився тільки у E2 і R5

**Де:** jtbd.md → висновок матриці; sitemap.md → екрани; research.html → PATH A / PATH B

**Канон:** петля завершується публікацією (виклав)

**Факт у jtbd.md → висновок матриці:**
```
→ Функції: вбудована бібліотека мемів + таймлайн + швидкий export (завершення петлі)
```
Export згаданий у дужках як "завершення петлі" — вже вторинний.

**Факт у sitemap.md → екрани:**
```
4. Екран експорту [Main, R5, E2]
```
Export-екран tagged як Main ✓ — але він четвертий в переліку і описується через R5 (Денис) і E2 (urgency), не через саму петлю.

**Факт у research.html → Глибина (PATH A / PATH B):**
- PATH A завершується кроком 4: "GIF on timeline — R3 completed" — петля НЕ ЗАМКНЕНА
- PATH B завершується кроком 3: "вставлено — R3 + R4 done ✓" — петля НЕ ЗАМКНЕНА
- Export є у PATH D — але відірваний як окремий path

**Наслідок:** в ілюстрації глибини Main job візуально завершується на вставці мему. Export — окремий сценарій Дениса. Читач сприймає: "зробив мем-відео = вставив мем". "Виклав" = опційна деталь.

**Правка потрібна в:**
- PATH A → додати кроки 5–6 (Експорт + Зберегти) зі статусом "Main job completed"
- PATH B → те саме
- PATH D перестає бути "окремим" і стає продовженням PATH A/B

## Зміщення 2 — R3 підмінив Main job як фокус продукту

**Де:** jtbd.md → висновок матриці; sitemap.md → навігаційна модель; personas.md → болі Ані

**Канон:** R3 — *один із кроків усередині* Main job, не головний job

**Факт у jtbd.md → висновок матриці:**
```
3 jobs у ядро MVP:
1. Main job
2. R1 — emoji/mood search
3. R3 — inline picker
```
— Правильно виглядає структурно. Але в тексті читається "інлайн пікер = ключова функція", а не "швидкий повний цикл до публікації".

**Факт у sitemap.md → навігаційна модель, Contextual:**
```
[Додати мем] toolbar кнопка | R3 | Editor + є ≥1 кліп
«Вставити мем не виходячи з монтажу.»
```
Кнопка "[Додати мем]" — фізичний початок другого кроку петлі "зняв → **додав мем** → виклав". Але вона tagged тільки R3. Main = відсутній.

**Факт у personas.md → болі Ані:**
```
App-switching — CapCut не має мем-бібліотеки... Додатковий аппсвіч на кожен мем.
```
Больовий контекст сформульований навколо моменту вставки, не навколо всього циклу "тригер → публікація".

**Наслідок:** продуктові рішення зміщуються на оптимізацію picker-кроку (R3), а не на оптимізацію всього циклу до публікації. Можна зробити ідеальний picker і при цьому мати friction на Export — Main job не виконаний.

**Правка потрібна в:**
- sitemap.md → "[Додати мем]" → додати Main до тегів, пояснити що це вхід у крок 2 петлі
- personas.md → болі Ані → розширити: pain = не тільки app-switching, а затримка між тригером і публікацією

## Зміщення 3 — "додав мем" деградувало до "вставив мем"

**Де:** jtbd.md → висновок матриці; sitemap.md → Сутності; research.html → PATH A

**Канон:** "додав мем" = знайшов + вибрав + розмістив (user-intent з контекстом)

**Деградація по фазах:**

| Фаза | Формулювання |
|---|---|
| Main job (jtbd.md) | "зняв → **додав мем** → виклав" |
| CLAUDE.md | "петля 'зняв → **додав мем** → виклав'" |
| jtbd.md висновок матриці | "зняв → **вставив** → виклав" ← перша деградація |
| R3 формулювання | "**вставити** мем не виходячи з монтажу" |
| sitemap.md МемОверлей | "R3 — '**вставити** мем не виходячи з монтажу'" |
| sitemap.md навігація | "[Додати мем] кнопка | R3 | '**Вставити** мем...'" |
| research.html PATH A | "GIF on timeline — **R3 completed**" |

**"Вставити"** = технічний жест (drag-and-drop, tap to place).
**"Додати мем"** = намір із контекстом: знайти відповідний мем за настроєм, вибрати, розмістити.

**Наслідок:** "вставити" підштовхує дизайн до оптимізації дії розміщення (позиціонування, тайм-код) замість оптимізації всього вибіркового досвіду (пошук, discovery, Recents).

**Правка потрібна в:**
- jtbd.md → висновок матриці: "зняв → **додав мем** → виклав" (не "вставив")
- Скрізь де "вставити мем" замінює "додати мем" як опис наміру — перевірити і вирівняти

## Зміщення 4 — Мем-пікер відв'язаний від Main job у тегуванні

**Де:** sitemap.md → екрани; research.html → картки екранів

**Канон:** Мем-пікер = серце диференціатора, пряме тіло кроку "додав мем" з Main job

**Факт у sitemap.md → екрани:**
```
3. Редактор [Main, R3, E1, E2]
   └─ Мем-пікер (bottom sheet) [R1, R2, R3, R4]  ← Main відсутній
```
Редактор в цілому tagged як Main. Але Мем-пікер — тільки R1–R4.

**Факт у research.html → картка екрану Редактора:**
```
Jobs: Main R3 E1 E2
Мем-пікер: R1 R2 R3 R4
```
Мем-пікер всередині Редактора — ті самі R1–R4 без Main.

**Проблема:** Мем-пікер = єдиний компонент що реалізує унікальну цінність "бібліотека вбудована в редактор". Він — фізичне вирішення Gap G1. Не позначити його Main = приховати зв'язок між диференціатором і головним job.

**Правка потрібна в:**
- sitemap.md → Мем-пікер → додати Main до тегів
- research.html → картка Редактора → Мем-пікер → Main + R1 R2 R3 R4

## Зміщення 5 — В моделі даних немає сутності що завершує Main job

**Де:** sitemap.md → Сутності

**Канон:** Main job завершується коли файл ВИЙШОВ на платформу

**Факт у sitemap.md → Сутності:**
```
1. Проект (Project) — контейнер петлі
2. Відео-кліп (VideoClip) — крок "зняв"
3. GIF / Мем — об'єкт для пошуку і вставки
4. МемОверлей (MemeOverlay) — результат кроку "додав мем"
5. Нещодавні GIF — cache для R4
```
В моделі НЕМАЄ сутності що представляє "виклав": ExportJob / OutputFile / PublishedClip або хоча б запис результату Export.

**МемОверлей described як:**
```
Main — "додав мем": МемОверлей — результат другого кроку петлі "зняв → додав мем → виклав"
```
МемОверлей описується як результат "Main job" — але він результат тільки кроку 2 з 3. Крок 3 ("виклав") в Сутностях не представлений взагалі.

**Наслідок:** в моделі даних "зробити відео" технічно завершується на МемОверлеї. Export не має свого Data Object. Це може призвести до того, що Export буде спроектований без належного UX-статусу (no confirmation screen, no result state).

**Правка потрібна в:**
- sitemap.md → Сутності → додати сутність "Результат Експорту" або позначити відсутність явно як відкрите питання
- Опис МемОверлею: він результат КРОКУ 2, не всього Main job

## Зміщення 6 — Export тегований як Денис-job, а не як завершення Main job

**Де:** jtbd.md матриця; sitemap.md екрани; research.html PATH D

**Канон:** Export = обов'язкове завершення Main job для ВСІХ персон

**Факт у jtbd.md матриці:**
```
R5 Експортувати у форматі платформи:
Аня — 3 (E2 вимагає export як фізичне завершення urgency-петлі)
Денис — 3 (job #3 "9:16 або 1:1")
Ліза — 2 (TikTok → 9:16)
```
R5 як job — визнаний важливим для всіх ✓. Але в personas.md jobs Ані:
```
Job 4: "Викласти відео якнайшвидше після завершення монтажу"
```
Це Job 4, найменш деталізований. Job 1 = мем-відео поки тренд, Job 2 = знайти мем, Job 3 = не перемикатись між аппами. Export — четвертий.

**Факт у sitemap.md навігаційна модель:**
```
[Кнопка Експорт] (header право) | R5 + E2 | Editor, якщо є ≥1 кліп
```
Кнопка Експорт tagged R5 + E2, Main — відсутній.

**Факт у research.html PATH D:**
```
Персони: Денис, Аня
```
PATH D в ілюстрації = "сценарій Дениса". PATH A/B = "сценарії Ані". Але Export — завершення Main job для Ані не менше ніж для Дениса.

**Наслідок:** коли команда думає "кому потрібен export" — відповідь звучить "Денису". Ані export сприймається як implied, не як explicit критична точка її job. Це ризик при пріоритизації: якщо Денис secondary, то R5 може зсунутись у backlog.

**Правка потрібна в:**
- sitemap.md → кнопка Експорт → додати Main до тегів
- research.html PATH D → зробити Аню першою персоною (вона primary, і для неї Export = E2 = "встигнути поки тренд живий")
- Переформулювати PATH D як продовження PATH A/B, не окремий сценарій

## Зміщення 7 — «мем або тренд» звузилось до «GIF-мем»

**Де:** Main job → R1–R4 → sitemap.md Сутності → Мем-пікер таби

**Канон:** "мем **або тренд** у стрічці" — тригер широкий, включає звукові тренди, формат-тренди, челенджі

**Деградація:**
| Фаза | Що є | Що зникло |
|---|---|---|
| Main job | «мем або тренд» — тригер широкий | — |
| R1–R4 (jtbd.md) | «мем», «GIF» | Звуковий тренд, формат-тренд |
| Сутності (sitemap.md) | GIF/Мем, МемОверлей | Немає AudioTrend, SoundOverlay |
| Мем-пікер таби | Trending GIF, Emoji search GIF, Recents | Trending Sound, Trending Format |

**Наслідок:** TikTok-тренди на 50–70% рухаються звуком. Юзер що «побачила тренд у стрічці» може хотіти додати звук, а не GIF. Продукт не покриває — але важливо зафіксувати це як scope decision, а не незнання.

**Рішення потрібне:** або уточнити Main job як «мем (GIF/стікер) у стрічці» — звузити до реального скоупу; або зберегти «мем або тренд» і відкрити питання AudioTrack як майбутнього step.

## Зміщення 8 — Г2 підтверджена, але живе в «Гіпотезах»

**Де:** jtbd.md → секція «Гіпотези»

**Факт:** Г2 (no-login) має статус «✓ Підтверджено для MVP». Секція «Гіпотези» описується: «Jobs що логічно випливають з контексту, але **не підкріплені даними ресерчу**». Протиріччя в одному файлі.

**Наслідок:** Читач бачить «Гіпотеза» → вважає «не підтверджено» → недооцінює no-login як стратегічну перевагу. Реально Г2 підтверджена аудитом 2026-06-14 + Reddit + SaleCycle + Experian — одне з найміцніших рішень MVP.

**Правка:** jtbd.md → або перенести Г2 до Related jobs з формулюванням «спробувати без реєстрації», або додати примітку в заголовок секції «Гіпотези» про Г2 як виключення. Г1 і Г3 — дійсно гіпотези.
